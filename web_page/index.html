03 — Memory Map

STM32F103 Memory Layout

The Cortex-M3 uses a 4 GB flat address space. Each segment below is animated into view — hover to explore addresses and uses.

Private Peripheral Bus 0xE000_0000 512 MB
0xE000_0000 – 0xFFFF_FFFF
Cortex-M3 internal peripherals: NVIC, SysTick, DWT, ITM, FPB, ROM table.
External Device / FSMC 0xA000_0000 1 GB
0xA000_0000 – 0xDFFF_FFFF
External devices mapped via FSMC: NOR Flash, NAND, PSRAM.
Peripheral Registers 0x4000_0000 512 MB · APB1/APB2/AHB
0x4000_0000 – 0x5FFF_FFFF
All on-chip peripherals: GPIO, USART, SPI, I2C, ADC, DMA, RCC, CRC. Each peripheral = 1 KB block.
SRAM 0x2000_0000 20 KB · Stack / Heap
0x2000_0000 – 0x2000_4FFF
Stack grows ↓ from top. Heap grows ↑. .data and .bss sections loaded from Flash on startup. Bit-banding enabled.
Flash Memory 0x0800_0000 64 KB · Vector Table + Code
0x0800_0000 – 0x0800_FFFF
Vector table at base, then .text (code), .rodata (constants). 128 × 512-byte erasable pages.
System Memory (Bootloader) 0x1FFF_F000 2 KB · ST Factory ROM
0x1FFF_F000 – 0x1FFF_F7FF
ST factory bootloader. Activate: BOOT0=1, BOOT1=0. Programs via USART1 or USB DFU.

Flash Memory — 64 KB

Starts at 0x08000000 and spans 64 KB (128 × 512-byte pages). The first bytes hold the Vector Table — the initial stack pointer and reset handler address. Code executes directly from Flash at 72 MHz via the prefetch queue.

SRAM — 20 KB

Starts at 0x20000000. At reset, startup code copies initialized variables from Flash (.data) and zeroes the BSS section. Stack grows down from 0x20005000, heap grows upward. Supports Cortex-M3 bit-banding for atomic bit operations.

Peripheral Registers

All peripherals memory-mapped from 0x40000000. APB1 at the base (USART2, I2C, SPI2, CAN), then APB2 (GPIO, ADC, SPI1, USART1, TIM1) and AHB (DMA, RCC, FLASH interface). Each register block is 1 KB.

System Memory — ST Bootloader

2 KB ROM at 0x1FFFF000 containing ST's factory-programmed bootloader. Activated by BOOT0=1, BOOT1=0 before reset. Programs via USART1 (or USB DFU) — no external probe required.

04 — Development Workflow

How to Program the Blue Pill

From bare-metal HAL to the Arduino ecosystem — the Blue Pill supports a wide range of development environments.

main.c — HAL Library
C · HAL 
/* STM32F103C8T6 — Blink PC13 LED */
/* STM32CubeIDE + STM32 HAL       */
#include "stm32f1xx_hal.h"

int main(void) {
  HAL_Init();
  SystemClock_Config();  // 72 MHz via PLL

  // Enable GPIOC peripheral clock
  __HAL_RCC_GPIOC_CLK_ENABLE();

  GPIO_InitTypeDef gpio = {0};
  gpio.Pin   = GPIO_PIN_13;
  gpio.Mode  = GPIO_MODE_OUTPUT_PP;
  gpio.Speed = GPIO_SPEED_FREQ_LOW;
  HAL_GPIO_Init(GPIOC, &gpio);

  while (1) {
    // PC13 is active-LOW on Blue Pill
    HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
    HAL_Delay(500);
  }
}
main.c — Register-Level
C · Bare-Metal 
/* Register-level — no HAL overhead */
#include "stm32f103xb.h"

static void delay(volatile uint32_t n) {
  while (n--);
}

int main(void) {
  // Enable APB2 clock → GPIOC
  RCC->APB2ENR |= RCC_APB2ENR_IOPCEN;

  // PC13: output push-pull @ 2 MHz
  GPIOC->CRH &= ~(0xF << 20);
  GPIOC->CRH |=  (0x2 << 20);

  while (1) {
    GPIOC->BSRR = GPIO_BSRR_BS13; // Set
    delay(500000);
    GPIOC->BSRR = GPIO_BSRR_BR13; // Reset
    delay(500000);
  }
}

05 — Hardware Programmer

Programming with ST-Link V2

The ST-Link V2 is the official debug/programmer for all STM32 devices. It uses the 2-wire SWD (Serial Wire Debug) protocol — faster, simpler, and more reliable than UART bootloading.

1Install ST-Link Drivers

Download and install the ST-Link USB driver from ST's website. On Windows, the STSW-LINK009 package is required. On Linux/macOS, libusb handles it automatically.

  • Windows: Install STSW-LINK009 driver package
  • Linux: Add udev rules for vid:0483 pid:3748
  • macOS: No driver needed — uses built-in HID
  • Verify in Device Manager: "STMicroelectronics STLink dongle"

2Install STM32CubeProgrammer

ST's official flashing utility. Supports SWD, JTAG, UART, USB DFU. Works with all STM32 families. Alternatively use OpenOCD (free, open-source).

  • Download STM32CubeProgrammer from st.com
  • Or install OpenOCD: winget install openocd
  • STM32CubeIDE bundles its own programmer automatically
  • PlatformIO uses OpenOCD internally via stlink

3Boot Jumper Settings

For SWD programming the boot jumpers do NOT need to be changed — you can flash directly into Flash at 0x08000000 while running normally.

  • BOOT0 = 0, BOOT1 = 0 → Boot from Flash (normal)
  • SWD works in this mode — press Reset if needed
  • No jumper changes required unlike UART bootloader
  • Hold Reset while connecting if MCU is locked

4Flash via STM32CubeProgrammer

  • Open STM32CubeProgrammer
  • Select ST-LINK interface in top-right dropdown
  • Click Connect — MCU info appears if wiring is correct
  • Go to Erasing & Programming tab
  • Browse to your compiled .bin or .elf file
  • Set Start Address: 0x08000000
  • Check Verify programming
  • Click Start Programming → Done!

5Flash via Command Line (OpenOCD)

Terminal
Shell 
# Flash .bin file using OpenOCD + ST-Link
openocd \
  -f interface/stlink.cfg \
  -f target/stm32f1x.cfg \
  -c "program firmware.bin verify reset exit 0x08000000"

# Erase entire Flash chip
openocd \
  -f interface/stlink.cfg \
  -f target/stm32f1x.cfg \
  -c "init; halt; stm32f1x mass_erase 0; exit"

# Flash .elf file (address embedded inside)
openocd \
  -f interface/stlink.cfg \
  -f target/stm32f1x.cfg \
  -c "program firmware.elf verify reset exit"

6Live Debug with GDB

  • Start OpenOCD GDB server: openocd -f interface/stlink.cfg -f target/stm32f1x.cfg
  • Listens on localhost:3333 (GDB) and localhost:4444 (telnet)
  • In another terminal: arm-none-eabi-gdb firmware.elf
  • Connect: (gdb) target remote :3333
  • Set breakpoint: (gdb) break main
  • Run: (gdb) continue
  • STM32CubeIDE handles all this with a GUI debugger

⚠️ Common Issues & Fixes

🔴 "No target connected"

Check your wiring: SWDIO → PA13, SWCLK → PA14, GND connected. Some Blue Pill clones have the SWD pins swapped. Try pressing Reset while clicking Connect in CubeProgrammer.

🟠 Target reads all 0xFF (blank)

The MCU may have read protection enabled. In STM32CubeProgrammer → OB (Option Bytes) → Set RDP to Level 0 and Apply. This will erase Flash — save your code first!

🟠 ST-Link not detected in PC

Install the STSW-LINK009 driver (Windows). Unplug and replug the USB. Check Device Manager for "Unknown USB Device" — likely a driver issue or a clone ST-Link needing firmware update.

🟢 Update ST-Link Firmware

Open STM32CubeProgrammer → Help → Firmware Upgrade. Outdated ST-Link V2 firmware (especially on clones) is a very common cause of connection failures. Always update first.

🔴 "Access port error" on connection

The Blue Pill may be executing code that disables SWD (e.g., PA13/PA14 remapped to GPIO). Hold the RESET button, click Connect in CubeProgrammer, then release Reset.

🟢 Verify Correct Pinout

The ST-Link V2 clone (silver stick) has its own pinout printed on the label. Pin 1 is marked. Use the correct pins: VCC=1, GND=4 (or 20), SWDIO=7, SWCLK=9, RST=15 (optional).
